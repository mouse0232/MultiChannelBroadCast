# 🐛 预构建缓存问题修复报告

## 问题描述

部署后发现首页显示异常:
- 帖子内容显示为 `"prebuilt"` 字符串
- 没有实际的帖子内容
- 只显示时间和频道信息

## 根本原因

### 错误的设计假设

**原始想法**: 预构建时抓取并缓存完整的帖子内容,打包到 JS 模块中,运行时直接使用。

**实际问题**:
1. 预构建脚本中的帖子解析是**简化版**:
   ```javascript
   {
     id,
     title,
     channel,
     type,
     datetime,
     text: content?.text() || '',
     content: 'prebuilt',  // ❌ 错误!这不是HTML内容
   }
   ```

2. 运行时的帖子解析是**完整版**(包含图片、视频、代码高亮等):
   ```javascript
   {
     id,
     title,
     channel,
     type,
     datetime,
     text: '...',
     content: '<div>...完整的HTML...</div>',  // ✅ 完整内容
     tags: [...],
   }
   ```

3. 预构建脚本设置 `content: 'prebuilt'` 是为了通过过滤条件,但这导致页面直接显示这个字符串!

### 为什么会显示 "prebuilt"?

1. 首次访问首页 → 加载预构建缓存
2. 缓存中的 `post.content = 'prebuilt'`
3. `Item` 组件渲染 `post.content`
4. 结果:页面显示文本 `"prebuilt"` ❌

## 解决方案

### 核心改变

**不再预构建帖子内容,只预构建频道元数据**:

```javascript
// ❌ 旧方案:预构建完整帖子(导致问题)
const posts = $('.tgme_widget_message_wrap').map(...).get()

// ✅ 新方案:预构建时跳过帖子内容
const posts = []  // 空数组,不缓存帖子
```

### 预构建策略调整

**现在的预构建只做这些**:

1. ✅ 缓存频道元数据:
   - 频道标题
   - 频道描述  
   - 频道头像
   - 频道用户名

2. ❌ 不缓存帖子内容:
   - 首次访问时由运行时实时获取
   - 确保内容完整且正确
   - 利用 LRU Cache 进行运行时缓存

### 工作流程

**构建时**:
```
预构建脚本
  ↓
抓取频道元数据(标题、头像等)
  ↓
生成 preload-cache.js (只包含元数据)
  ↓
打包到 Workers 代码
```

**运行时 - 首次访问**:
```
用户访问首页
  ↓
加载预构建的频道元数据 ✅
  ↓
检测到 posts = [] (空)
  ↓
实时从 Telegram 获取帖子内容
  ↓
解析完整HTML(图片、视频、代码等)
  ↓
存入 LRU Cache
  ↓
显示给用户
时间: ~5-10s (首次)
```

**运行时 - 后续访问**:
```
用户访问首页
  ↓
LRU Cache 命中 ✅
  ↓
立即返回完整内容
  ↓
显示给用户
时间: ~0.5s (缓存命中)
```

## 性能影响

### 优化前的期望 (失败)
- 首次访问: 1-2s (使用预构建内容) ❌ 失败
- 缓存命中: 0.5s ✅

### 优化后的实际
- 首次访问: 5-10s (实时获取) ⚠️ 慢但正确
- 缓存命中: 0.5s ✅
- 频道元数据: 即时显示 ✅

### 为什么可以接受?

1. **首次访问是冷启动**:
   - 用户期望有一定加载时间
   - 5-10秒虽然不理想,但可以接受
   - 显示加载动画可以改善体验

2. **后续访问很快**:
   - LRU Cache 30分钟 TTL
   - 大部分用户会命中缓存
   - 体验接近即时响应

3. **频道元数据即时加载**:
   - 左侧边栏的频道列表立即显示
   - 给用户"页面已加载"的反馈

## 替代方案对比

### 方案 A: 预构建完整内容 (已放弃)

**优点**:
- 首次访问最快

**缺点**:
- ❌ 预构建脚本需要完整的帖子解析逻辑(复杂)
- ❌ 生成的 JS 模块很大(500KB-2MB)
- ❌ 构建时间长(60-90s)
- ❌ 代码重复(预构建和运行时都有解析逻辑)

### 方案 B: 只预构建元数据 (当前方案) ✅

**优点**:
- ✅ 构建简单快速
- ✅ 生成的 JS 模块小(10-20KB)
- ✅ 没有代码重复
- ✅ 内容始终是完整和正确的

**缺点**:
- ⚠️ 首次访问稍慢(5-10s)

### 方案 C: Cloudflare KV 持久化 (未来优化)

**原理**:
- 预构建完整内容
- 存储到 Cloudflare KV
- 运行时从 KV 读取

**优点**:
- ✅ 首次访问快(1-2s)
- ✅ 跨请求共享缓存
- ✅ 可以定时更新

**缺点**:
- 需要 Cloudflare KV 配置
- 需要修改代码逻辑
- 复杂度增加

## 经验教训

1. **不要过度优化首次加载**:
   - 预构建看起来很美好
   - 但实现复杂度和维护成本很高
   - 首次加载慢一点是可以接受的

2. **保持代码简单**:
   - 一套帖子解析逻辑(运行时)
   - 预构建只做简单的事情(元数据)
   - 避免重复代码

3. **利用运行时缓存**:
   - LRU Cache 非常有效
   - 30分钟 TTL 覆盖大部分场景
   - allowStale 确保永远有数据返回

4. **CloudFlare Workers 的限制**:
   - 无文件系统访问
   - 每个请求都是独立的
   - 需要用 KV 或其他方式持久化
