# 完善 Docker 部署方案 - 需求文档

**文档版本**: 1.0  
**创建日期**: 2026-05-21  
**状态**: 草案  

---

## 1. 引言

### 1.1 目的

本文档描述 Multi-Channel Broadcast 项目 Docker 部署方案的完善需求，旨在实现与 Cloudflare Workers 部署模式的功能对等。

### 1.2 范围

本项目当前的 Docker 部署仅支持 Astro SSR 前端，缺失后台异步抓取、定时任务、数据库等核心功能。本需求涵盖以下范围：

- 后端 API 服务实现
- 异步任务队列集成
- 定时任务调度器
- SQLite 数据库集成
- 媒体代理服务
- Docker 配置优化

### 1.3 定义与缩略语

| 术语 | 说明 |
|------|------|
| CF Workers | Cloudflare Workers 无服务器计算平台 |
| D1 | Cloudflare SQLite 数据库服务 |
| Queue | Cloudflare Queues 消息队列服务 |
| Docker | 容器化部署方案 |
| API | 应用程序编程接口 |

## 2. 总体描述

### 2.1 产品背景

Multi-Channel Broadcast 是一个多频道 Telegram 内容抓取与展示系统，目前主要部署在 Cloudflare Workers 平台，功能稳定。为实现私有化部署和离线运行，需要完善 Docker 部署方案。

### 2.2 当前问题

现有 Docker 部署存在以下功能缺失：

1. **缺失异步抓取能力**: Docker 版本无法后台运行抓取任务
2. **缺失定时调度**: 无法按照预设频率自动抓取新内容
3. **缺失数据持久化**: 没有数据库存储抓取的帖子数据
4. **缺失 API 服务**: 前端无法获取动态数据
5. **缺失关键词过滤**: 无法在 Docker 环境下执行过滤逻辑
6. **缺失推送服务**: 无法推送新内容到 Telegram

### 2.3 用户特征

**主要用户群体**:
- 个人开发者：希望在本地或私有服务器部署
- 小型团队：需要数据自主可控
- 网络受限用户：无法访问 Cloudflare 服务的地区

**技术能力**:
- 具备 Docker 基础使用能力
- 能够配置环境变量
- 能够执行基本的容器管理命令

## 3. 具体需求

### 3.1 功能性需求

#### 3.1.1 后端 API 服务

**FR-001: 帖子列表 API**
- **描述**: 系统应提供获取帖子列表的 API 端点
- **端点**: `GET /api/posts`
- **参数**: 
  - `channel` (可选): 频道过滤器
  - `limit` (可选): 返回数量限制，最大 100
  - `before` (可选): 游标分页，获取早于指定时间的帖子
  - `after` (可选): 游标分页，获取晚于指定时间的帖子
- **响应**: JSON 格式，包含帖子数组
- **优先级**: P0

**FR-002: 单个帖子 API**
- **描述**: 系统应提供获取单个帖子详情的 API
- **端点**: `GET /api/post/:id`
- **参数**: `id` (路径参数): 帖子 ID，格式为 `channel/messageId`
- **响应**: JSON 格式，包含帖子对象
- **优先级**: P0

**FR-003: 帖子搜索 API**
- **描述**: 系统应提供按关键词搜索帖子的 API
- **端点**: `GET /api/posts/search`
- **参数**: 
  - `q` (必需): 搜索关键词
  - `channel` (可选): 频道过滤器
  - `limit` (可选): 返回数量限制
- **响应**: JSON 格式，包含匹配的帖子数组
- **优先级**: P0

**FR-004: 频道列表 API**
- **描述**: 系统应提供获取频道列表的 API
- **端点**: `GET /api/channels`
- **响应**: JSON 格式，包含频道信息数组（含抓取进度）
- **优先级**: P0

**FR-005: 初始化 API**
- **描述**: 系统应提供手动触发全量抓取的 API
- **端点**: `GET /api/init`
- **安全**: 需验证 API Secret Key
- **功能**: 遍历所有配置频道，执行全量抓取
- **优先级**: P0

**FR-006: 重新抓取 API**
- **描述**: 系统应提供重新抓取并更新旧帖子的 API
- **端点**: `GET /api/regrab`
- **安全**: 需验证 API Secret Key
- **参数**: `limit` (可选): 每个频道重新抓取的数量
- **优先级**: P1

#### 3.1.2 异步任务队列

**FR-010: 任务队列服务**
- **描述**: 系统应实现异步任务队列，用于处理抓取任务
- **实现**: 使用 BullMQ + Redis 或内存模式
- **并发**: 支持并发处理多个频道抓取任务（默认 5 个）
- **优先级**: P0

**FR-011: 任务生产**
- **描述**: 定时调度器应将抓取任务发送到队列
- **触发**: 每分钟执行一次 Cron 任务
- **内容**: 每个频道生成一个独立任务
- **优先级**: P0

**FR-012: 任务消费**
- **描述**: 队列消费者应并行处理抓取任务
- **并发度**: 可配置（默认 5）
- **错误处理**: 失败任务应记录日志，可选择重试
- **优先级**: P0

#### 3.1.3 定时任务调度

**FR-020: Cron 调度器**
- **描述**: 系统应实现定时任务调度器
- **频率**: 每分钟执行一次（可配置）
- **功能**: 
  - 生成抓取任务并发送到队列
  - 定期清理过期数据（可选）
- **优先级**: P0

**FR-021: 数据清理**
- **描述**: 系统应定期清理过期数据
- **周期**: 每次 Cron 执行时检查
- **保留策略**: 保留最近 1 年的帖子
- **清理内容**: 
  - 过期帖子（published_at < 当前时间 - 1 年）
  - 孤立推送日志（post_id 不在 posts 表中）
- **优先级**: P1

#### 3.1.4 数据库集成

**FR-030: SQLite 数据库**
- **描述**: 系统应集成 SQLite 数据库用于数据持久化
- **表结构**: 
  - `posts` 表：存储帖子数据
  - `channel_meta` 表：存储频道元数据和抓取进度
  - `push_logs` 表：记录推送日志
- **优先级**: P0

**FR-031: 索引优化**
- **描述**: 系统应为查询频繁的字段创建索引
- **索引**: 
  - `posts(channel)`
  - `posts(published_at)`
  - `posts(channel, published_at)`
- **优先级**: P1

**FR-032: 数据持久化**
- **描述**: 数据库文件应持久化存储，容器重启不丢失
- **实现**: Docker Volume 挂载到宿主机
- **路径**: `/app/data/app.db`
- **优先级**: P0

#### 3.1.5 内容抓取

**FR-040: Telegram 抓取**
- **描述**: 系统应从 Telegram 频道抓取内容
- **源**: `t.me/s/{channel}` 公开页面
- **防风控**: 
  - 使用随机 User-Agent
  - 支持多个 Telegram Host 轮流请求
  - 请求失败自动重试
- **优先级**: P0

**FR-041: HTML 解析**
- **描述**: 系统应解析 Telegram HTML 页面提取帖子数据
- **提取字段**:
  - 帖子 ID（`channel/messageId`）
  - 频道名称
  - 标题（自动提取）
  - 内容（HTML 格式，含媒体）
  - 发布时间（ISO 8601）
- **优先级**: P0

**FR-042: 媒体处理**
- **描述**: 系统应处理帖子中的媒体资源
- **图片**: 使用 wsrv.nl 代理或本地缓存
- **视频/音频**: 使用本地 /static/ 代理，支持 Range 请求
- **优先级**: P0

#### 3.1.6 关键词过滤

**FR-050: 关键词过滤**
- **描述**: 系统应支持基于规则的关键词过滤
- **规则源**: `filter-rules.json` 配置文件
- **模式**: 
  - 黑名单模式：匹配到关键词则拦截
  - 白名单模式：未匹配关键词则拦截
- **开关**: 通过环境变量 `FILTER_ENABLED` 控制
- **优先级**: P0

**FR-051: 渠道级规则**
- **描述**: 系统应支持为不同频道配置独立的过滤规则
- **继承**: 可选择是否继承全局规则
- **优先级**: P1

#### 3.1.7 推送服务

**FR-060: Telegram 推送**
- **描述**: 系统应将新帖子推送到指定的 Telegram 频道
- **触发**: 抓取到新帖子后自动触发
- **去重**: 检查 push_logs 表防止重复推送
- **格式**: 
  - 有图：发送图文消息
  - 无图：发送纯文本消息
- **优先级**: P0

**FR-061: 推送配置**
- **描述**: 系统应支持通过环境变量配置推送
- **配置项**:
  - `TELEGRAM_PUSH_ENABLED`: 是否启用推送
  - `TELEGRAM_BOT_TOKEN`: Bot Token
  - `TELEGRAM_PUSH_CHANNEL_ID`: 目标频道 ID
- **优先级**: P0

**FR-062: 首次运行保护**
- **描述**: 系统应在首次运行时不触发推送，防止消息轰炸
- **判断**: 如果频道没有 last_msg_id 记录，视为首次运行
- **优先级**: P0

#### 3.1.8 媒体代理

**FR-070: 图片代理**
- **描述**: 系统应提供图片代理服务，解决国内访问问题
- **方案**: 
  - 默认：使用 wsrv.nl CDN
  - 可选：本地文件缓存
- **端点**: `/img-proxy?url={encoded_url}`
- **缓存**: 本地缓存应持久化存储
- **优先级**: P1

**FR-071: 视频代理**
- **描述**: 系统应提供视频代理服务，支持拖动进度条
- **端点**: `/static/{host}/file/{path}`
- **功能**: 透传 Range 请求头，支持 206 Partial Content
- **优先级**: P1

### 3.2 非功能性需求

#### 3.2.1 性能需求

**NFR-001: 响应时间**
- API 端点响应时间应小于 100ms（缓存命中）
- API 端点响应时间应小于 500ms（数据库查询）
- 页面加载时间应小于 2 秒

**NFR-002: 并发能力**
- 系统应支持同时处理至少 5 个频道的抓取任务
- API 服务应支持至少 10 个并发请求

**NFR-003: 资源消耗**
- 内存占用应小于 1GB
- CPU 占用应小于 100%（闲时）

#### 3.2.2 可靠性需求

**NFR-010: 数据持久化**
- 数据库应在容器重启后保持数据完整
- 抓取进度不应丢失

**NFR-011: 错误恢复**
- 抓取失败应记录日志并继续处理下一个任务
- 关键操作失败应有重试机制

**NFR-012: 健康检查**
- 系统应提供健康检查端点
- Docker 应配置健康检查，失败时自动重启

#### 3.2.3 安全性需求

**NFR-020: API 认证**
- 管理接口（/api/init, /api/regrab）应验证 API Secret Key
- 敏感操作日志应记录但不输出敏感信息

**NFR-021: 凭据保护**
- Bot Token 应通过环境变量传入，不写入代码
- `.env` 文件应加入 `.gitignore`

**NFR-022: 代理安全**
- 图片代理应限制只能代理 Telegram CDN 域名
- 防止被用作开放代理（Open Proxy）

#### 3.2.4 可维护性需求

**NFR-030: 日志记录**
- 系统应记录关键操作的日志
- 日志应包含时间戳、操作类型、结果状态
- 支持日志轮转，避免磁盘占用过高

**NFR-031: 监控告警**
- 系统应提供健康检查端点
- 长时间未抓取应记录告警

**NFR-032: 备份恢复**
- 应提供数据库备份脚本
- 支持从备份恢复数据

#### 3.2.5 兼容性需求

**NFR-040: 环境兼容**
- 系统应在 Linux x86_64 和 ARM64 架构上运行
- 支持 Docker Engine 20.10+

**NFR-041: API 兼容**
- API 端点应保持与 CF Workers 版本兼容
- 前端调用无需修改

### 3.3 接口需求

#### 3.3.1 环境变量

**IR-001: 核心配置**
- `CHANNELS`: 逗号分隔的频道列表
- `SITE_NAME`: 站点名称
- `LOCALE`: 语言设置
- `TIMEZONE`: 时区设置

**IR-002: 抓取配置**
- `TELEGRAM_HOST`: Telegram 主机列表（逗号分隔）
- `FILTER_ENABLED`: 是否启用关键词过滤

**IR-003: 推送配置**
- `TELEGRAM_PUSH_ENABLED`: 是否启用推送
- `TELEGRAM_BOT_TOKEN`: Bot Token
- `TELEGRAM_PUSH_CHANNEL_ID`: 目标频道

**IR-004: Docker 配置**
- `DOCKER`: 固定值 `true`
- `DATA_DIR`: 数据目录（默认 `/app/data`）
- `CACHE_DIR`: 缓存目录（默认 `/app/cache`）
- `HOST`: 监听地址（默认 `0.0.0.0`）
- `PORT`: 监听端口（默认 `4321`）

**IR-005: 安全配置**
- `API_SECRET_KEY`: API 密钥（管理接口验证）
- `API_LOGGING_ENABLED`: 是否启用 API 请求日志

#### 3.3.2 文件接口

**IR-010: filter-rules.json**
- 描述：关键词过滤规则配置文件
- 位置：项目根目录
- 格式：JSON
- 结构：
  ```json
  {
    "global": {
      "mode": "blacklist|whitelist",
      "rules": [
        { "id": "1", "pattern": "关键词", "ruleType": "keyword|regex", "isActive": true }
      ]
    },
    "channels": {
      "channel1": {
        "mode": "blacklist|whitelist",
        "inheritGlobal": true,
        "rules": []
      }
    }
  }
  ```

#### 3.3.3 Docker 接口

**IR-020: 端口映射**
- 容器端口：4321
- 宿主机端口：可配置（默认 4321）

**IR-021: 数据卷**
- `/app/data`: 数据库持久化
- `/app/cache`: 媒体缓存

## 6. 风险与缓解

### 6.1 技术风险

**R-001: Telegram 风控**
- 风险：频繁抓取导致 IP 被封禁
- 缓解：
  - 使用多 Host 轮流请求
  - 设置随机延迟
  - 使用随机 User-Agent

**R-002: 数据库性能**
- 风险：SQLite 并发性能不足
- 缓解：
  - 启用 WAL 模式
  - 优化索引
  - 数据量过大时建议迁移到 PostgreSQL

### 6.2 运维风险

**R-010: 数据丢失**
- 风险：容器删除导致数据丢失
- 缓解：
  - 强制使用 Docker Volume 持久化
  - 提供备份脚本
  - 文档明确说明数据目录

**R-011: 服务宕机**
- 风险：服务异常无人知晓
- 缓解：
  - 配置健康检查
  - 提供日志查询方式
  - 建议配置外部监控
