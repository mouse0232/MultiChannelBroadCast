# 关键词过滤功能需求文档

**版本**: 5.0  
**创建日期**: 2026-05-19  
**更新日期**: 2026-05-19  
**状态**: 草案

---

## 1. 概述

### 1.1 项目背景

Multi-Channel Broadcast 是一个聚合多个 Telegram 频道到微博的项目。
后端使用 **Cloudflare Workers** (`workers/cache-worker.js`) 通过 Cron + Queue 异步抓取 Telegram 频道内容，存入 D1 数据库。

### 1.2 问题陈述

当前采集是**无过滤采集**，所有频道内容都会被抓取和存储。这导致：
- 垃圾广告、违规内容混入数据库
- 无法针对特定频道定制采集策略
- 推送消息包含不相关内容

### 1.3 需求目标

增加关键词过滤功能，支持：
1. 关键词匹配和正则表达式匹配
2. 白名单 (Include) 和黑名单 (Exclude) 两种模式
3. 按渠道分别配置，支持继承全局规则
4. **过滤规则通过 JSON 配置文件管理**，打包进 Worker，修改后重新部署生效

### 1.4 术语定义

| 术语 | 定义 |
|------|------|
| 白名单 (Include) | 仅允许匹配规则的帖子通过 |
| 黑名单 (Exclude) | 阻止匹配规则的帖子通过 |
| 渠道 (Channel) | Telegram 频道用户名 (如 `channelname`) |
| 帖子 (Post) | Telegram 单条消息 (包含标题、内容、媒体) |
| 规则 (Rule) | 单个过滤条件 (关键词或正则表达式) |

## 2. 需求规格 (EARS 模式)

### 2.1 功能性需求

#### FR-001: JSON 配置文件
**ID**: FR-001  
**描述**: 系统应使用 JSON 配置文件定义过滤规则  
**优先级**: P0  
**验收标准**:
- 配置文件为 `filter-rules.json`，打包进 Worker 代码
- 配置文件符合预定义的 JSON Schema
- 配置文件支持 UTF-8 编码 (中文、英文、Emoji)

#### FR-002: 白名单模式
**ID**: FR-002  
**描述**: 当配置白名单模式时，系统应仅允许标题或内容包含指定关键词/匹配正则的帖子通过  
**优先级**: P0  
**验收标准**:
- 白名单模式下，帖子必须包含至少一个白名单关键词才能采集
- 白名单为空时，该渠道所有帖子均不采集
- 匹配不区分大小写
- 同时支持关键词和正则表达式

#### FR-003: 黑名单模式
**ID**: FR-003  
**描述**: 当配置黑名单模式时，系统应阻止标题或内容包含指定关键词/匹配正则的帖子存入 D1  
**优先级**: P0  
**验收标准**:
- 黑名单模式下，帖子包含任一黑名单关键词时被拦截
- 黑名单为空时，该渠道所有帖子均采集 (不排除其他过滤条件)
- 匹配不区分大小写
- 同时支持关键词和正则表达式

#### FR-004: 渠道独立配置
**ID**: FR-004  
**描述**: 系统应为每个 Telegram 频道提供独立的过滤规则配置  
**优先级**: P0  
**验收标准**:
- 支持按渠道 ID (Telegram 用户名) 分别配置规则集
- 不同渠道可使用不同的过滤模式 (白名单/黑名单)
- 支持为特定渠道添加特有规则

#### FR-005: 规则继承
**ID**: FR-005  
**描述**: 渠道规则应支持继承全局规则并添加特有规则  
**优先级**: P1  
**验收标准**:
- 未明确配置规则的渠道自动继承全局规则
- 渠道可在继承基础上添加额外规则
- `inheritGlobal` 字段控制是否继承 (默认 `true`)
- 渠道特有规则与继承规则合并后统一匹配

#### FR-006: 采集流程集成
**ID**: FR-006  
**描述**: 系统应在 Queue Consumer 处理帖子时执行过滤  
**优先级**: P0  
**验收标准**:
- 在 `parsePosts` 解析后、写入 D1 之前执行过滤
- 被过滤的帖子不存入 `posts` 表
- 被过滤的帖子不触发 Telegram 推送
- 被编辑的帖子同样需要过滤检测

#### FR-007: 正则表达式支持
**ID**: FR-007  
**描述**: 系统应支持使用正则表达式进行高级模式匹配  
**优先级**: P1  
**验收标准**:
- 规则类型字段区分 `keyword` 和 `regex`
- 正则表达式使用 JavaScript RegExp 引擎
- 支持常用元字符 (`.`, `*`, `+`, `?`, `[]`, `()`, `|`)
- 正则表达式编译失败时跳过该规则并记录日志

#### FR-008: 环境变量开关
**ID**: FR-008  
**描述**: 系统应支持通过环境变量控制过滤功能的启用/禁用  
**优先级**: P0  
**验收标准**:
- 环境变量 `FILTER_ENABLED` 控制过滤功能开关
- 默认值为 `false` (默认不启用过滤)
- 设置为 `true` 时启用过滤
- 设置为 `false` 时跳过过滤逻辑，直接采集所有内容

### 2.2 非功能性需求

#### NFR-001: 性能
**ID**: NFR-001  
**描述**: 过滤操作不应显著影响采集性能  
**优先级**: P0  
**验收标准**:
- 单个帖子过滤耗时 < 5ms
- 规则缓存到内存，减少重复解析
- Queue Consumer 批次处理能力不受影响 (5 个/批次)

#### NFR-002: 可靠性与容错
**ID**: NFR-002  
**描述**: 过滤功能故障不应影响主采集流程，需对 JSON 配置错误进行容错处理  
**优先级**: P0  
**验收标准**:
- 配置文件不存在时，降级为"不过滤"模式并记录日志
- **JSON 格式错误时，降级为"不过滤"模式，记录 ERROR 日志，不中断采集**
- **JSON 语法错误 (如缺少逗号、括号不匹配) 不得导致 Worker 启动失败**
- 正则表达式编译失败时，跳过该规则并记录错误
- 不因过滤逻辑错误而误拦截正常帖子

#### NFR-003: 可维护性
**ID**: NFR-003  
**描述**: 过滤规则应易于维护和版本追踪  
**优先级**: P1  
**验收标准**:
- JSON 配置文件可 Git 版本控制
- 规则包含 `createdAt` 时间戳
- 提供配置示例和文档说明

#### NFR-004: 资源限制
**ID**: NFR-004  
**描述**: 过滤功能应符合 Cloudflare Workers 资源限制  
**优先级**: P0  
**验收标准**:
- 内存占用 < 10MB (规则缓存)
- CPU 时间 < 50ms/请求 (过滤逻辑)

## 3. 业务规则

### 3.1 优先级规则
- 黑名单规则优先级 > 白名单规则 (当同时配置时)
- 渠道特有规则优先级 = 继承的全局规则 (合并后统一匹配)
- 正则表达式规则优先级 = 关键词规则 (按配置顺序匹配)

### 3.2 匹配规则
- 所有匹配不区分大小写 (`i` 标志)
- 支持 UTF-8 多语言字符 (中文、英文、Emoji、俄文等)
- 关键词匹配：子字符串匹配 (`content.includes(keyword)`)
- 正则匹配：`regex.test(content)`
- 标题和内容任一匹配即视为匹配成功

### 3.3 过滤时机
- 过滤发生在 `parsePosts` 解析之后
- 过滤发生在写入 D1 之前
- 过滤发生在 Telegram 推送触发之前
- 被编辑的帖子同样需要过滤检测

## 4. 配置文件设计

### 4.1 文件位置

```
project-root/
└── filter-rules.json           # 过滤规则配置文件 (打包进 Worker)
```

### 4.2 配置格式

```json
{
  "global": {
    "mode": "blacklist",
    "rules": [
      {
        "id": "1",
        "pattern": "垃圾广告",
        "ruleType": "keyword",
        "isActive": true,
        "description": "过滤垃圾广告",
        "createdAt": "2026-05-19T00:00:00.000Z"
      },
      {
        "id": "2",
        "pattern": "spam|advertisement",
        "ruleType": "regex",
        "isActive": true,
        "description": "过滤英文广告",
        "createdAt": "2026-05-19T00:00:00.000Z"
      }
    ]
  },
  "channels": {
    "channel1": {
      "mode": "blacklist",
      "inheritGlobal": true,
      "rules": [
        {
          "id": "3",
          "pattern": "特定关键词",
          "ruleType": "keyword",
          "isActive": true,
          "description": "",
          "createdAt": "2026-05-19T00:00:00.000Z"
        }
      ]
    }
  }
}
```

### 4.3 配置字段说明

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `global.mode` | string | 是 | 全局过滤模式：`blacklist` 或 `whitelist` |
| `global.rules` | array | 是 | 全局规则列表 |
| `channels.{id}.mode` | string | 否 | 渠道过滤模式，默认继承全局 |
| `channels.{id}.inheritGlobal` | boolean | 否 | 是否继承全局规则，默认 `true` |
| `channels.{id}.rules` | array | 否 | 渠道特有规则列表 |
| `rules[].id` | string | 是 | 规则唯一 ID (**手动或自动生成**) |
| `rules[].pattern` | string | 是 | 关键词或正则表达式 |
| `rules[].ruleType` | string | 是 | `keyword` 或 `regex` |
| `rules[].isActive` | boolean | 是 | 是否启用 (默认 `true`) |
| `rules[].description` | string | 否 | 规则描述 (可选) |
| `rules[].createdAt` | string | - | 创建时间 |

### 4.4 空配置模板

```json
{
  "global": {
    "mode": "blacklist",
    "rules": []
  },
  "channels": {}
}
```

## 5. 验收测试用例

| 用例 ID | 场景 | 输入 | 预期输出 |
|--------|------|------|----------|
| AT-001 | 黑名单关键词匹配 | 帖子内容包含"垃圾广告" | 不存储，不推送 |
| AT-002 | 白名单关键词匹配 | 帖子内容包含"工作邮件" | 存储，可推送 |
| AT-003 | 白名单不匹配 | 帖子内容不包含任何白名单词 | 不存储 |
| AT-004 | 正则匹配 | 内容匹配 `spam\|advertisement` | 黑名单模式下拦截 |
| AT-005 | 渠道继承 | 渠道未配置规则集 | 使用全局规则过滤 |
| AT-006 | 渠道独立 | 渠道配置 `inheritGlobal: false` | 仅使用渠道特有规则 |
| AT-007 | 规则更新 | 修改配置文件并重新部署 | 下次采集生效 |
| AT-008 | 性能测试 | 100 条规则，100 个帖子 | 过滤总耗时 < 500ms |
| AT-009 | 降级处理 | 配置文件加载失败 | 降级为"不过滤"模式，记录告警 |
| AT-010 | 正则错误处理 | 无效正则表达式 | 跳过该规则，记录错误日志 |
| AT-011 | 开关控制 | `FILTER_ENABLED=false` | 跳过过滤逻辑，全部采集 |
